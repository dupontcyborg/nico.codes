## Code blocks

Here's a TypeScript snippet showing how numpy-ts handles disposable arrays — the `using` keyword ensures WASM memory gets freed at scope exit:

```typescript
import { multiply, add, sin, cos } from "numpy-ts/core";

function buildField(X: NDArray, Y: NDArray, drift: number) {
  using t0 = multiply(Y, 2.2);
  using t1 = add(t0, drift * 7.5);
  using t2 = sin(t1);
  using t3 = multiply(X, 3.4);
  using t4 = add(t3, drift * 4.8);
  using t5 = cos(t4);
  using t6 = multiply(t5, 0.62);
  
  // t0–t6 are automatically freed here
  return add(t2, t6);
}
```

And here's something smaller inline: the `configureWasm({ maxMemory: 32 * 1024 * 1024 })` call sets a 32 MiB ceiling on the WASM heap.

A short Python snippet for comparison:

```python
import numpy as np

def build_field(X, Y, drift):
    u = np.sin(Y * 2.2 + drift * 7.5) + np.cos(X * 3.4 + drift * 4.8) * 0.62
    v = np.cos(X * 2.35 - drift * 7.2) + np.sin(Y * 2.55 + drift * 3.7) * 0.5
    magnitude = np.sqrt(u**2 + v**2 + 1e-6)
    return u / magnitude, v / magnitude
```

## A longer paragraph

When you're building a flow field renderer that runs at 60fps in the browser, every allocation matters. JavaScript's garbage collector is generous — too generous, if you're working with typed arrays backed by WebAssembly memory. A single leaked intermediate array per frame means the heap is exhausted in under ten seconds. This is why numpy-ts uses TypeScript 5.2's explicit resource management (`using` declarations) to deterministically free WASM-backed arrays at scope boundaries. It's not glamorous, but it's the difference between a smooth animation and a tab crash.

The trick is making the API feel natural despite the manual memory management happening underneath. Nobody wants to call `.dispose()` on every temporary. The `using` keyword makes it read almost like Python — you declare your intermediates, do your math, and the runtime handles cleanup when the block exits. It's the closest JavaScript gets to RAII without actually having destructors.
